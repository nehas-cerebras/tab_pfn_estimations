Feature Distribution Embedder — why the two phases?

The data is a grid: R rows × C columns, each cell embedded to a 128-dim vector.

The one-sentence answer: One step needs a whole column at a time (all rows), the next step needs a whole row at a time (all columns). Those two demands pull in opposite directions, so no single way of slicing the grid satisfies both — that's why the work is split into two passes.

STAGE 1 Feature Distribution Embedder — works down a column

Each column is processed independently. For one column it does two things:

• block1 reads all R rows of that column and squeezes them into a tiny fixed 128×128 summary ("what does this feature's distribution look like?").

• block2 then gives every row a new embedding by looking at that summary.

Key point: to build the summary, block1 must see the entire column at once. Columns never talk to each other here.

⚔️ The conflict

Stage 1's block1 wants to slice the grid into columns (it needs all rows of a column together).

Stage 2 wants to slice the grid into rows (it needs all columns of a row together).

These are the two perpendicular directions of the same grid. You cannot cut the grid one way and satisfy both. The overlapping cell below is claimed by both.

Your idea vs. the two-phase design

❌ Slice by columns only

Do a few columns at a time through Stage 1 — peak compute memory is small, true. But Stage 2 needs whole rows, so you must keep every column's output until all columns are done. The full grid piles up in memory before Stage 2 can even start.

Ends up holding the entire (R × C × 128) grid ≈ 51.2 GB.

✅ Two phases

→

Phase 1 (left): slice by columns, run block1 over all rows, keep only the tiny summaries (≈10 MB). Rows are then thrown away.
Phase 2 (right): slice by rows in bands. Each band flows through block2 and Stage 2 together, reading the cached summaries, and immediately collapses C→4. The big grid is never all in memory at once.

Peak ≈ one row-band + summaries ≈ well under 1 GB.

The tiny summary from Phase 1 is the bridge: it lets Phase 2 give each row an embedding without needing the original rows block1 consumed — which is also exactly the little cache you keep for predicting 15 days later.
